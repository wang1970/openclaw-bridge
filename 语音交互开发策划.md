# 语音交互开发策划

## 需求背景
实现语音实时监听桥接工具：持续监听麦克风 → 唤醒词检测 → 语音转文字 → 提交给 OpenClaw Agent。
通过 `小龙虾` 唤醒词触发，实现自然的语音交互体验。

---

## 第一版目标：端到端验证 ✅ 已完成

### 实现功能
1. 持续监听麦克风音频流
2. **唤醒词检测**："小龙虾"，KWS 专用引擎（主要）+ ASR 文本匹配（后备）
3. VAD 检测语音起止（含预缓冲防丢开头、尾部静音裁剪）
4. sherpa-onnx ASR 离线转文字
5. 通过 **openclaw agent CLI** 提交文本给 OpenClaw Agent
6. 命令行启动，打印日志

### 验证结果（2026-03-27）
端到端全链路实测通过：
```
17:52:09  KWS 唤醒 "小龙虾"
17:52:12  ASR 识别 "你在不在"
17:52:14  提交到 OpenClaw (openclaw agent CLI)
17:52:20  AI 响应: "在呢！猫哥，小龙虾二世随时待命 🦞 有啥事直接说就行～"
```
全链路延迟约 11 秒（其中 Agent 响应约 6 秒，属于 OpenClaw 侧延迟）。

## 第四版：Gateway WebSocket 通信 ✅ 已完成

### 问题
CLI 子进程每次调用有 ~6s 启动开销，Agent 响应总延迟高。

### 方案
逆向 OpenClaw Gateway WebSocket 协议，实现 Ed25519 设备签名认证。

### 实现细节
- **协议流程**：challenge → connect（含 Ed25519 签名）→ hello-ok → chat.send → 流式接收
- **设备签名**：`deviceId = hex(SHA-256(publicKey))`，签名内容 `v2|deviceId|clientId|clientMode|role|scopes|signedAtMs|token|nonce`
- **关键常量**：`client.id = "clawdbot-control-ui"`，`client.mode = "webchat"`
- **消息发送**：`chat.send`，参数 `{sessionKey: "agent:main:main", message, idempotencyKey}`
- **流式响应**：`agent` 事件（stream=assistant/lifecycle/tool/end）+ `chat` 事件（state=delta/final）
- **密钥持久化**：`.device_key.json` 文件，自动生成 Ed25519 密钥对
- **依赖**：`websockets`（async）+ `cryptography`（Ed25519）
- **CLI 降级**：WebSocket 不可用时自动降级为 CLI 模式

### 效果
- 认证成功即预热完成（无需 CLI 预热），启动到就绪 < 1 秒
- Agent 响应延迟从 CLI 的 6s+ 降到约 2s

## 第五版：交互策略优化 ✅ 已完成

### 核心问题
V4 的交互策略过于简单，每次都要重新唤醒，无法连续对话，用户体验差。

### 设计参考
对标小爱同学的交互模式：
- 唤醒后有明确的等待状态
- 回答后保持侦听，支持直接追问（多轮对话）
- 句子间自然停顿不会截断
- 等回复期间的新消息智能缓冲

### 状态机（五态）

```
                    唤醒"小龙虾"                 用户说话(4s内拼接)
┌──────────┐ ─────────────────→ ┌──────────────┐ ─────────────→ ┌──────────────┐
│   IDLE    │                   │WAKE_DETECTED │                 │LISTENING_CMD  │
│ 空闲监听  │ ←── 冷却/超时 ─── │ 等待命令      │ ←── 跳过唤醒词 ─│ 录制命令      │
└──────────┘                   └──────────────┘                 └──────┬───────┘
     ↑                                │                                │
     │                        CMD_TIMEOUT(15s)               CMD_END_SILENCE(4s)
     │                        播报"小龙虾跑了"                       │
     │                                │                                ▼
     │                         ┌──────────────┐              ┌──────────────────┐
     │                         │    IDLE       │              │ _finish_command  │
     │                         └──────────────┘              │ 拼接 → 发送/缓冲  │
     │                                                       └───────┬──────────┘
     │                                                               │
     │  FOLLOWUP_TIMEOUT(8s)                                          │
     │  或收到唤醒词                                                   │
     │         ┌───────────────────────────────────────────────────────┘
     │         │
     │         ▼
     │  ┌──────────────────┐     回复到达      ┌──────────┐     TTS播报     ┌──────────┐
     └──│   FOLLOWUP        │ ──────────────→ │  PLAYING  │ ────────────→ │ FOLLOWUP  │
        │ 多轮后续侦听(8s)  │                   │ TTS播放中 │               │ 可直接追问 │
        │ 用户可直接说话    │ ←────────────── │ 麦克风暂停│               │          │
        └──────────────────┘     FOLLOWUP     └──────────┘               └──────────┘
```

### 智能缓冲策略
解决"发送了也没用"的问题（Agent 不能很好地处理并发消息）：

| 场景 | 行为 |
|------|------|
| Agent 回复快 + 用户追问 | 正常流程，追问直接发送 |
| Agent 回复慢 + 用户说完了 | 缓冲新消息，等回复到了→播报→立刻发缓冲 |
| 缓冲期间又有新话 | 覆盖缓冲（只保留最新一条） |

### 关键参数（config.py）

| 参数 | 值 | 说明 |
|------|----|------|
| `WAKE_COOLDOWN` | 3.0s | 唤醒冷却，所有状态统一检查 |
| `CMD_TIMEOUT` | 15.0s | 唤醒后等待第一句话的超时 |
| `CMD_END_SILENCE` | 4.0s | 命令录制中，停顿多久算说完（> 自然句间停顿 ~2s） |
| `FOLLOWUP_TIMEOUT` | 8.0s | 回答后保持侦听，可直接追问 |
| `TTS_LENGTH_SCALE` | 0.7 | TTS 语速（0.5~2.0） |

### 部署配置（config.py）

以下参数部署到外部 Hub 时需由用户自行填写：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `OPENCLAW_URL` | `http://127.0.0.1:18789/` | 网关地址（外部部署改为对应域名/IP） |
| `OPENCLAW_TOKEN` | `106d...` | WebSocket 认证 Token |
| `OPENCLAW_SESSION` | `voice-bridge` | 会话标识 |
| `OPENCLAW_SESSION_KEY` | `agent:main:main` | Agent 通道 Key（对应前端 session 参数） |
| `VOICE_BRIDGE_TAG` | `[bridge] ` | 消息前缀标识（Agent 据此切换人格） |

### 语音桥消息标识
每条发给 Agent 的消息前自动拼接 `VOICE_BRIDGE_TAG` 前缀（默认 `[bridge] `），Agent 端通过该前缀识别消息来源为语音桥，从而切换到语音交互人格。
- WS 通道和 CLI 备用通道均支持
- 设为空字符串可关闭前缀

### 其他改进
- **心跳过滤**：`HEARTBEAT_OK` 等系统消息不触发 TTS
- **ASR 异常保护**：空音频片段导致 onnxruntime 崩溃 → try/except 静默跳过
- **唤醒提示音**：唤醒成功后 TTS 短促反馈（预留，待实现更好的提示音方案）

## 线程架构：双线程生产者-消费者

```
┌─────────────────────┐    事件队列       ┌─────────────────────┐
│   音频采集线程        │   (Queue)         │   策略处理线程        │
│   (Producer)        │ ──────────────→   │   (Consumer)        │
│                     │                   │                     │
│ · 麦克风持续采集      │                   │ · 从队列取事件        │
│ · KWS 唤醒词检测      │ ─ WakeEvent      │ · KWS唤醒 → 等命令   │
│ · VAD 能量检测       │ ─ UtteranceEvent │ · ASR 识别转文字      │
│ · 按音量分割语音片段   │ ─ AgentRespEvent │ · ASR文本匹配唤醒(后备)│
│ · 产出语音段+唤醒事件  │                   │ · 五态状态机管理      │
│ · 无状态，不关心内容   │                   │ · 命令拼接与提交      │
│ · 极轻量，不阻塞      │                   │ · 多轮后续侦听       │
│                     │                   │ · 智能缓冲策略       │
└─────────────────────┘                   └─────────────────────┘
                                                    │
                                                    ▼
                                          ┌──────────────────┐
                                          │  OpenClawClient   │
                                          │  WebSocket 主通道  │
                                          │  CLI 降级备用      │
                                          └──────────────────┘
```

**设计原则：**
- 音频线程**只管分段+检测**：VAD 按 RMS 阈值切割语音片段，KWS 实时流式检测唤醒词
- 策略线程**只管决策**：从队列取事件做 ASR/状态判断/命令提交
- 两个线程通过 `queue.Queue` 通信，音频线程永不阻塞
- 状态机只在策略线程中运行，无需加锁

## OpenClaw 客户端：WebSocket 主通道

```
┌─────────────────────────────────────────────┐
│            OpenClawClient                    │
│                                             │
│  ┌───────────────────────────────────┐      │
│  │ WebSocket 主通道（默认）           │      │
│  │ · websockets (asyncio)            │      │
│  │ · Ed25519 设备签名认证             │      │
│  │ · challenge → connect → hello-ok  │      │
│  │ · chat.send 发送消息              │      │
│  │ · 流式接收 Agent 响应              │      │
│  │ · 自动重连（5s 间隔）             │      │
│  │ · 认证成功即预热完成              │      │
│  └───────────────────────────────────┘      │
│                                             │
│  ┌───────────────────────────────────┐      │
│  │ CLI 备用通道                       │      │
│  │ · websockets/cryptography 未安装  │      │
│  │ · openclaw agent --message        │      │
│  │ · 独立线程异步执行                 │      │
│  └───────────────────────────────────┘      │
└─────────────────────────────────────────────┘
```

### WebSocket 协议详情
- **连接地址**：`ws://127.0.0.1:18789`（从 `OPENCLAW_URL` 转换）
- **设备密钥**：`.device_key.json`，Ed25519 密钥对持久化
- **签名格式**：`v2|{deviceId}|clawdbot-control-ui|webchat|operator|scopes|{signedAtMs}|{token}|{nonce}`
- **deviceId**：`hex(SHA-256(publicKey))`
- **sessionKey**：`agent:main:main`
- **响应事件**：`agent`（stream=assistant/lifecycle/tool/end）+ `chat`（state=delta/final）
- **心跳过滤**：`HEARTBEAT_OK` 等系统消息自动跳过

## 使用方法

```bash
# 1. 首次运行，下载模型
python download_model.py

# 2. 安装依赖
pip install -r requirements.txt

# 3. 确保 OpenClaw Gateway 运行中
openclaw gateway

# 4. 启动语音桥接
python main.py

# 5. 说出 "小龙虾" 唤醒，然后说出指令
# 6. 回答后可直接追问，无需再次唤醒（8秒内）
```

### 命令行参数
```
--model-dir DIR       ASR 模型目录
--openclaw-url URL    OpenClaw 网关地址（默认 http://127.0.0.1:18789/）
--no-kws              禁用 KWS，仅用 ASR 文本匹配唤醒词
```

## 已知问题 & 待办

### 已解决
- ~~HTTP POST 405~~ → OpenClaw 仅接受 WebSocket
- ~~WebSocket 认证 invalid handshake~~ → 去除 URL 末尾斜杠
- ~~WebSocket 认证 invalid request frame~~ → Ed25519 设备签名协议
- ~~唤醒词回声（KWS+VAD 同时截取）~~ → 策略线程跳过仅含唤醒词的片段
- ~~Windows GBK 编码 emoji 崩溃~~ → subprocess encoding="utf-8"
- ~~TTS emoji 正则吞掉中文字符~~ → 修正 Unicode 范围
- ~~TTS 文本清洗~~ → `clean_text_for_tts()` 统一处理
- ~~单字识别噪音~~ → 长度 ≤ 1 丢弃
- ~~预热阻塞启动~~ → 后台线程 / WS 认证即就绪
- ~~唤醒超时无反馈~~ → 播报"小龙虾跑了"
- ~~TTS max_num_sentences=1 截断~~ → 改为 10
- ~~CLI ~6s 启动开销~~ → WebSocket 长连接替代
- ~~Agent 回复太短（英文）~~ → sessionKey 改为 agent:main:main
- ~~HEARTBEAT_OK 被 TTS 播报~~ → 心跳消息过滤
- ~~ASR 空音频崩溃~~ → try/except 保护
- ~~每次都要重新唤醒~~ → FOLLOWUP 多轮后续侦听
- ~~命令发送太快截断用户说话~~ → CMD_END_SILENCE=4s + 智能缓冲
- ~~等回复期间发送浪费~~ → 回复到了再发缓冲消息
- ~~空闲状态非唤醒词日志泄露~~ → 先判断匹配再打印日志
- ~~缓冲状态 IDLE 超时残留~~ → `_goto_idle` 清除 `_waiting_response` 和 `_pending_cmd`
